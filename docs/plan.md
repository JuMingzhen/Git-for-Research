# Git for Research 学术协作系统 MVP 规划

## Summary

把这个项目定义为一个**面向导师-学生课题组的学术协作 Web 系统**。  
第一版重点解决两类问题：

1. 日常进度不透明，导师难以持续低成本了解学生状态。
2. 组会前后信息断裂，会前背景不足、会后任务回流效率低。

因此，第一版产品主线定为：

- 以**日常进度管理**为主线
- 集成**组会协作**能力
- 使用**文本输入**，不做语音/图片等多模态
- AI 做**轻量辅助**：总结、任务拆分、分支提示、周报整理、会议问答
- 先给未来多 Agent 协作**预留接口**，但不在 MVP 中真正实现 Agent 参与工作流

## Product Design

### 1. 核心产品定位

一句话定义：

**一个把“科研进展管理 + 组会协作 + AI 轻量辅助”合在一起的 Research OS。**

它不是传统项目管理工具的简单翻版，而是专门贴合学术场景：

- 项目不是简单任务列表，而是**研究主线**
- 学生不是普通成员，而是**自己分支的负责人**
- 组会不是普通会议，而是**研究过程中的关键节点**
- AI 不是替代人决策，而是帮你**减轻记录、整理、同步信息的负担**

### 2. 用户角色与权限

第一版固定两类角色：

- `Advisor`
  - 创建课题项目
  - 查看所有学生分支
  - 创建组会节点
  - 查看会前摘要、会后纪要、任务拆分结果
  - 对学生分支给反馈

- `Student`
  - 更新自己的研究分支
  - 创建子分支/子任务
  - 提交阶段进展、问题、下一步计划
  - 查看与自己相关的会议任务和历史记录

不做多层级权限。博士/硕士/本科生统一先按 `Student` 处理，后续再扩展。

### 3. 研究分支模型

这是产品最重要的概念，要做成系统主骨架。

#### 项目结构

- 一个课题对应一个 `Project`
- 每个项目有一个 `Main Branch`
- 每个学生默认有一个自己的 `Personal Branch`
- 学生可以在自己的分支下继续创建 `Sub-branches`

#### 语义解释

- `Main Branch`
  - 表示整个课题组共同关注的研究主线
- `Personal Branch`
  - 表示某个学生当前负责的研究方向
- `Sub-branch`
  - 表示该学生为了推进研究拆出的子问题、实验线、文献线、写作线等

#### 设计原则

系统里的“分支”不是 Git 那种技术分支，而是**研究工作流分支**。  
它的作用是帮助用户把抽象研究过程变成可追踪的结构。

### 4. 组会节点模型

第一版把组会定义为主分支上的一种特殊节点 `Meeting Node`。

每次组会包含：

- 会前材料收集
- AI 自动生成会前摘要
- 会中记录
- 会后 AI 生成会议纪要
- AI 将讨论结果拆分到各学生分支
- 会后支持问答检索

这意味着组会不是孤立页面，而是会对整个项目状态产生影响。

### 5. 第一版必须做出的 AI 能力

AI 只做“轻辅助”，但要做得有感知价值。

#### AI 功能 1：进展总结

输入：

- 学生最近若干条更新
- 当前分支状态

输出：

- 本阶段在做什么
- 已完成什么
- 目前问题是什么
- 下一步建议

#### AI 功能 2：子任务/子分支提示

当学生更新研究进度时，AI 给出建议，例如：

- 这项工作可以拆成实验验证、文献补充、结果分析三个子分支
- 你当前卡在方法选择上，可以单独开一个“方案比较”子分支

这是“科研版代码补全”的雏形。

#### AI 功能 3：会前 briefing

在组会开始前，系统自动为导师生成：

- 每位学生近一周进展概况
- 当前卡点
- 需要重点追问的问题
- 哪些分支长时间没有更新

#### AI 功能 4：会后纪要与任务拆分

会后基于会议记录生成：

- 会议摘要
- 每位学生的下一步任务
- 对应更新到各自分支

#### AI 功能 5：会议问答

用户可以基于历史组会文本提问，例如：

- “上次组会导师对我实验设计的意见是什么？”
- “这个项目之前有没有讨论过换 baseline？”
- “王同学本周负责什么？”

实现上先用简单的检索增强问答，不追求复杂知识图谱。

#### AI 功能 6：周报整理

系统自动把学生一周零散更新整理成周报草稿，学生确认后提交。

## System Design

### 1. 技术路线

为了兼顾易做、易演示、后续可扩展，第一版采用：

- 前端：`Next.js + TypeScript`
- 后端：`FastAPI`
- 数据库：`PostgreSQL`
- AI 接口层：统一封装一个 `LLM Service`
- 检索问答：`pgvector` 或等价向量检索方案
- 认证：简单邮箱/邀请码登录，先不做复杂组织系统

这样选的原因：

- Web 原型最适合 demo 展示
- 前后端分离后，未来接 Agent、移动端、多模态都更容易
- FastAPI 很适合做 AI 功能编排
- PostgreSQL 足够稳，也方便后面加向量检索

### 2. 核心数据对象

第一版的数据模型固定如下：

- `User`
  - id, name, role, team_id

- `Team`
  - id, name

- `Project`
  - id, title, description, owner_id, status

- `ResearchBranch`
  - id, project_id, parent_branch_id, owner_id, title, goal, status

- `ProgressUpdate`
  - id, branch_id, author_id, content, blockers, next_step, created_at

- `Meeting`
  - id, project_id, title, scheduled_at, raw_notes, ai_summary

- `MeetingTask`
  - id, meeting_id, assignee_id, branch_id, description, due_hint, status

- `WeeklyDigest`
  - id, user_id, project_id, week_range, draft_content, confirmed_content

- `DocumentChunk`
  - id, source_type, source_id, text, embedding
  - 用于会议问答检索

### 3. 页面与功能范围

第一版 Web demo 做 6 个页面就够：

1. `项目总览页`
   - 查看项目主线
   - 查看成员状态
   - 查看最近组会和异常分支

2. `学生分支页`
   - 查看某学生主分支
   - 查看其子分支结构
   - 查看最近更新和任务

3. `进展更新页`
   - 学生提交本次进展、问题、下一步
   - AI 生成摘要与子任务建议

4. `组会页`
   - 创建组会
   - 导入会议文本记录
   - 查看 AI 会前 briefing 和会后纪要

5. `任务视图页`
   - 按学生、按分支查看会后任务
   - 支持状态更新

6. `问答页`
   - 针对历史会议与进展记录提问
   - 返回答案和引用来源

### 4. 后端接口边界

第一版 API 只保留最关键的一组：

- `POST /projects`
- `GET /projects/:id`
- `POST /branches`
- `GET /branches/:id`
- `POST /updates`
- `GET /branches/:id/updates`
- `POST /meetings`
- `POST /meetings/:id/summarize`
- `POST /meetings/:id/split-tasks`
- `POST /weekly-digests/generate`
- `POST /qa/ask`

AI 层内部再拆为：

- `summarize_progress(updates, branch_context)`
- `suggest_subbranches(update_text, branch_context)`
- `build_pre_meeting_brief(project_context, recent_updates)`
- `summarize_meeting(raw_notes, project_context)`
- `extract_meeting_tasks(meeting_summary, participants)`
- `answer_from_history(question, retrieved_chunks)`

### 5. Agent 扩展接口预留

第一版不实现多 Agent，但必须预留接口。

预留方式：

- 所有“执行者”统一抽象成 `Actor`
- 当前 `Actor` 只支持 `HumanUser`
- 后续新增 `AgentUser` 时，不重写主流程，只扩展 actor 类型
- 任务分配、分支拥有者、会议参与者都以 actor_id 关联

这样以后才能把“智能体作为研究成员”平滑接进去。

## MVP Demo Plan

### 阶段 0：项目提案与原型定义

产出物：

- 一份项目提案文档
- 一张系统流程图
- 一份 MVP 功能边界清单

目标：

- 把“要做什么”和“先不做什么”讲清楚
- 能向导师/同学解释这是个学术协作系统，而不是泛项目管理工具

### 阶段 1：最小可演示版本

必须完成：

- 导师创建项目
- 学生分支结构展示
- 学生提交进展
- AI 生成进展总结
- 组会记录导入
- AI 生成会前摘要与会后纪要
- AI 拆分任务到学生分支
- 简单的历史问答

这一阶段的 demo 已经能完整演示：

“学生平时更新 -> 导师会前查看 -> 开组会 -> 会后任务自动回流”

### 阶段 2：可用性增强

可加入：

- 周报自动整理
- 更新模板
- 长时间未更新提醒
- 分支状态看板
- 引用来源更清晰的问答结果

### 阶段 3：未来扩展

后续再做：

- 语音输入与会议转录
- 文献/PDF 接入
- 图表、实验结果、多模态附件
- Agent 自动执行部分研究辅助工作
- 与 GitHub、Notion、飞书等系统联动

## Demo Scenario

第一版演示脚本固定为一个典型课题组流程：

1. 导师创建一个课题项目，例如“多模态科研助手”
2. 两个学生分别拥有自己的研究分支
3. 学生 A 提交本周进展，AI 自动总结并建议拆出两个子分支
4. 学生 B 提交遇到的问题，系统标记为 blocker
5. 导师在组会前打开项目，看到 AI 会前 briefing
6. 组会结束后导入会议记录文本
7. AI 生成会议纪要，并把任务拆分给学生 A 和 B
8. 学生打开各自分支，看到新增任务
9. 用户在问答页提问“上次组会导师让学生 A 下周完成什么？”
10. 系统返回答案并引用相关会议内容

## Test Plan

### 功能测试

1. 导师可以创建项目并看到主分支
2. 学生可以拥有自己的分支并新增子分支
3. 学生提交更新后，AI 能生成结构化总结
4. 组会文本导入后，系统能生成纪要
5. 会议任务能正确关联到具体学生和分支
6. 历史问答能返回可读答案，并带来源片段

### 场景测试

1. 一个学生连续三天更新进度，系统能生成一份清晰周报
2. 导师在组会前无需逐条读记录，也能快速知道每位学生状态
3. 会议中提出的新任务能在会后自动回到个人分支
4. 用户忘记某次会议内容时，问答功能可以查到关键结论

### 风险测试

1. AI 总结过于空泛时，是否能保留原文回看
2. AI 拆任务出错时，是否允许人工编辑
3. 问答回答不准时，是否显示引用来源避免误导
4. 更新文本很短或很乱时，系统是否仍能工作

## Assumptions

- 第一版只面向**导师-学生型课题组**
- 第一版只做**Web demo**
- 输入全部为**文本**，不实现语音和图片
- AI 只做**轻量辅助**，不做自主科研代理
- 问答基于**历史文本检索**，不做复杂知识图谱
- 权限模型保持简单，不做学院级、实验室级复杂组织管理
- 第一版优先验证“学术协作是否更顺畅”，而不是追求技术炫技
- 如果时间有限，优先级顺序是：
  1. 进展更新与分支结构
  2. 组会纪要与任务拆分
  3. 历史问答
  4. 周报整理
  5. Agent 扩展接口
